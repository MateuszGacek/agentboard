# 08 — Internationalization

## Required languages

```txt
en — English
pl — Polish
cs — Czech
```

## Scope

All visible UI text must use translation keys.

This includes:

- navigation,
- buttons,
- forms,
- validation messages,
- empty states,
- toasts,
- error messages,
- dashboard labels,
- task properties,
- AI feature text,
- theme/language switch labels.

Seeded demo task content may remain English for MVP.

## Language resolution

Order:

1. authenticated user preference,
2. localStorage preference,
3. browser language,
4. fallback `en`.

Browser mapping:

```txt
pl-* -> pl
cs-* -> cs
en-* -> en
other -> en
```

## File structure

Recommended:

```txt
apps/web/src/i18n/
  index.ts
  config.ts
  locales/
    en/common.json
    pl/common.json
    cs/common.json
```

If translations grow, split by namespace:

```txt
auth.json
app.json
board.json
task.json
dashboard.json
ai.json
settings.json
```

For MVP, `common.json` is acceptable if well organized.

## Translation key naming

Use stable, semantic keys:

```txt
auth.login.title
app.nav.dashboard
board.createTask
board.wip.exceeded
task.detail.title
task.ai.improveButton
dashboard.metrics.overdue
settings.theme.system
```

Avoid keys based on current English text.

## Language switcher

Required behavior:

- visible in app shell,
- accessible on mobile menu,
- animated but subtle,
- switches without full page reload,
- stores preference locally,
- if logged in, optionally persists preference to user profile.

UI labels:

```txt
EN
PL
CS
```

Expanded labels for accessibility:

```txt
English
Polski
Čeština
```

## Date and number formatting

Use `Intl.DateTimeFormat` and selected language.

Examples:

```ts
new Intl.DateTimeFormat(locale, { dateStyle: "medium" }).format(date);
```

Due-date states should be translated:

- Overdue,
- Today,
- This week,
- No due date.

## Validation messages

Validation should be translatable.

Example keys:

```txt
validation.required
validation.email
validation.minLength
validation.maxLength
validation.invalidDate
```

Zod errors can map to translation keys instead of hardcoded English.

## Czech quality note

Czech translations should be simple and product-like. Avoid overly literal Polish-to-Czech wording.

Preferred tone:

- concise,
- professional,
- SaaS/product style.

## Example keys

```json
{
  "app": {
    "name": "AgentBoard",
    "tagline": "AI Kanban for software teams shipping with agents"
  },
  "nav": {
    "dashboard": "Dashboard",
    "board": "Board",
    "projects": "Projects",
    "settings": "Settings"
  },
  "theme": {
    "light": "Light",
    "dark": "Dark",
    "system": "System"
  },
  "language": {
    "en": "English",
    "pl": "Polish",
    "cs": "Czech"
  }
}
```

## Acceptance criteria

- No visible hardcoded UI text in React components.
- Language can be switched at runtime.
- Browser language detection works.
- Fallback to English works.
- EN/PL/CS files have the same keys.
- Long Polish/Czech strings do not break mobile layout.
- Date formatting follows selected locale.
